# 502 Preview Error - Root Cause Analysis

## Date: 2026-03-30

## Problem
User sees `HTTP ERROR 502` when accessing `3000-ujfsjfujxkviovju.daytonaproxy01.net`.

## Investigation Findings

### 1. Sandbox ID Mismatch
- The preview URL uses the subdomain `ujfsjfujxkviovju` — this is NOT the sandbox ID.
- The actual sandbox ID is `11b43c4e-b983-49b8-a96d-c4cf3f12e418`.
- The URL `3000-ujfsjfujxkviovju.daytonaproxy01.net` is a **signed/proxy preview URL** generated by `sandbox.getSignedPreviewUrl(3000, 3600)`.
- These signed URLs **expire after 1 hour** (3600 seconds).

### 2. Sandbox State Was Stopped
- The sandbox was in `"state": "stopped"` state with `autoStopInterval: 15` minutes.
- The sandbox auto-stops after 15 minutes of inactivity.
- After auto-stop/restart, all running processes (like `npm run dev`) are killed.

### 3. No Worker or Project Files Found
- After starting the sandbox back up, `/home/daytona/` contains only default dotfiles.
- No `generation-worker.mjs`, no `worker.log`, no `website-project/` directory.
- No Node.js processes running, nothing on port 3000.
- This means the sandbox was **rebuilt/cleaned** after the auto-stop timeout, or the agent's session expired and files were lost.

### 4. The 502 Chain
```
User clicks preview → 3000-ujfsjfujxkviovju.daytonaproxy01.net
→ Daytona proxy routes to sandbox:3000
→ Sandbox is stopped (or nothing on port 3000)
→ 502 Bad Gateway
```

## Root Causes (Multiple)

1. **Signed preview URL expiration**: URLs generated by `getSignedPreviewUrl(3000, 3600)` expire after 1 hour. After that, they always 502.

2. **Sandbox auto-stop**: With `autoStopInterval: 15` (minutes), the sandbox stops after 15 min of idle. When stopped, no dev server is running. The dev server is started via `npm run dev &` which is a background process — it dies when the sandbox stops.

3. **No persistence on restart**: When the sandbox restarts (either through SDK `start()` or Daytona's own wake-up), the dev server is NOT automatically restarted. The worker script and project files appear to have been lost (possibly due to non-persistent storage or sandbox recreation).

## Decisions Made

### Decision 1: Use non-expiring preview URLs
Instead of `getSignedPreviewUrl(3000, 3600)`, use `getPreviewLink(3000)` which generates a standard proxy URL that works as long as the sandbox is running. The signed URL adds a layer of complexity and expiration that isn't needed since the sandbox already has `public: false`.

### Decision 2: Add sandbox keep-alive / auto-restart mechanism  
When a user loads a project page with a sandbox_id, the frontend should check if the sandbox is running and wake it up if needed. The dev server should be restarted as part of this wake-up.

### Decision 3: Increase autoStopInterval for active projects
Set `autoStopInterval` to at least 60 minutes (or 0 to disable) so sandboxes don't die while users are actively looking at the preview.

### Decision 4: Store project files on persistent volume or use sandbox snapshots
The current approach of running everything in ephemeral sandbox storage means project files can be lost. Consider using Daytona volumes or snapshots to persist the generated project.

---

## Changes Implemented

### 1. `app/api/generate-daytona/route.ts`
- **Sandbox creation**: Added `autoStopInterval: 60` (up from default 15 min)
- **System prompt**: Changed launch step from `npm run dev &` to `npx vite --host 0.0.0.0 --port 3000 &` with extra rule emphasizing host binding
- **Preview URL**: Switched from `getSignedPreviewUrl(3000, 3600)` (expires 1hr) to `getPreviewLink(3000)` (stable URL) with signed URL as fallback at 2hr expiry

### 2. `app/api/restart-server/route.ts`
- **Complete rewrite**: Removed `child_process.spawn()` approach (broken on Vercel serverless)
- Now uses Daytona SDK directly: wakes stopped sandboxes, kills port 3000, restarts Vite with `--host 0.0.0.0`, gets fresh preview URL, updates DB
